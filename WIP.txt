How to resume on a fresh machine
--------------------------------

1. Clone with submodules:

       git clone --recurse-submodules git@github.com:avwohl/dosemu.git
       cd dosemu

2. Install build deps:

       sudo apt install -y build-essential cmake ninja-build meson \
           pkg-config libsdl2-dev libsdl2-net-dev libpng-dev \
           libopusfile-dev libspeexdsp-dev libfluidsynth-dev \
           libslirp-dev libasound2-dev libxi-dev libglib2.0-dev \
           patch nasm p7zip-full

3. Install optional tools for compiling test fixtures:

       sudo apt install -y bcc bin86 binutils-djgpp
       sudo snap install --edge open-watcom

4. Build:

       make         # applies dosbox patch, runs meson+ninja, builds dosemu

   First build takes ~5 minutes for dosbox-staging; incremental is fast.

5. Verify:

       build/dosemu tests/HELLO.COM        # prints dosemu-hello-ok
       build/dosemu tests/DPMI_STAGE5.COM  # prints dpmi-stage5=hello-from-pm
       build/dosemu tests/EXE2BIN.EXE      # prints Open Watcom banner

   CI runs ~15 fixtures on every push via .github/workflows/ci.yml.

DPMI stage status
-----------------

    Stage 1 ✓  INT 31h denial stub (CF=1 / AX=8001h)
    Stage 2 ✓  INT 2Fh/1687h reports DPMI present with failing entry (flipped
               to working entry in stage 3)
    Stage 3 ✓  Real → PM switch (both 16-bit AX=0 and 32-bit AX=1).
               Builds GDT at seg 0x1800, patches stacked CS to PM selector,
               CPU_SET_CRX(0, CR0_PE), loads PM selectors.
    Stage 4 ✓  Full subset shipped: AX=0400 (get DPMI version),
               AX=0006/0007 (get/set seg base, GDT or LDT), AX=0000
               (alloc N LDT descriptors), AX=0001 (free), AX=0002
               (segment-to-descriptor with repeat-call cache), AX=0003
               (selector increment=8), AX=0204/0205 (get/set real-mode
               IVT), AX=0500 (get free memory info).  LDT lives at
               0x1B00 (256 slots, installed via GDT[6]+LLDT).  INT 31h
               has IDT gates for both 16-bit and 32-bit PM.  INT 2Fh/
               1687h advertises 32-bit DPMI (BX=1).
    Stage 5 ✓  INT 21h from 16-bit PM reflects to host via IDT interrupt
               gate at 0x1A00 + GDT selector 0x28 for CB_SEG.  Handler uses
               SegPhys() so addressing works in either mode.
    Stage 5'  ✓ 32-bit PM end-to-end: DPMI_STAGE5_32.COM enters PM with
               AX=1, executes 32-bit-encoded code, does INT 21h through a
               32-bit IDT gate, and exits via AH=4C.  Required a second
               INT 21h callback installed with CB_IRETD (`66 CF`) so the
               stub unwinds the 12-byte frame a 32-bit gate pushes.
    Stage 6 ✓  PM linear memory alloc: AX=0501 (allocate), AX=0502
               (free), AX=0503 (resize in place -- shrink always, grow
               iff adjacent tail is free).  No relocation on grow.
    Stage 7 ✗  DJGPP hello.exe end-to-end. Stage 4 (subset) + Stage 6
               (minimal) both in place -- next real-client attempt is
               the closest it has ever been.


Current DPMI limitations
------------------------

1. Only INT 21h is reflected.  Other INT vectors in PM fault through an
   uninitialised IDT slot.  Fixtures and clients must CLI before the switch
   to mask IRQ0 (timer interrupt → INT 8 → fault).

2. DOS4GW binaries (what Open Watcom's `owcc-dos4g` produces) don't use
   DPMI directly -- they spawn DOS4GW.EXE via AH=4Bh.  So they also need
   AH=4Bh load-and-execute before they'll work.

3. DJGPP hello.exe needs stages 4 and 6 (LDT + PM memory alloc) before it
   can complete startup.  This is the cleaner next target than DOS4GW.

Next-session picks
------------------

Pick one of these as the opening move:

A. Stage 4 minimum: AX=0400 (version), AX=0006/0007 (get/set seg base).
   Enough for many DPMI clients to initialise without allocating
   descriptors.

B. Stage 6 minimum: AX=0501 returning an MCB-allocated region as a
   32-bit linear address.

C. (done 2026-04-21) 32-bit PM fixture DPMI_STAGE5_32.COM exercises
   the 32-bit interrupt-gate path end-to-end and is wired into CI.
   While building it, uncovered + fixed that a 16-bit IRET stub can't
   unwind the 12-byte frame a 32-bit gate pushes -- added a CB_IRETD
   callback for the 32-bit path.

D. Skip DPMI work temporarily: AH=4Bh load-and-execute so DOS4GW
   binaries can spawn their extender child.  Medium-large; needs CPU
   state save/restore across recursive DOSBOX_RunMachine calls and
   child PSP setup.

E. Find a non-spawning DPMI client.  Open Watcom ships DOS32A which is
   an alternative extender; might be self-contained.  Also a tiny
   standalone DJGPP program (built elsewhere) linked against CWSDPMI
   stub would be ideal.

Recommended order: A + B (real INT 31h services) → E or built-DJGPP
→ iterate.

DOS32A probe (2026-04-21 late afternoon)
----------------------------------------

~/ow/binw/dos32a.exe loads as MZ, completes its real-mode init, does
the DPMI switch (with AX=1 -- enters 32-bit PM directly), then faults
inside PM at an un-installed IDT slot:
"INT:Gate Selector points to illegal descriptor with type 0x0".

After shipping 16-bit PM reflection (every CB_SEG-pointing IVT entry
now has a matching PM IDT gate), DOS32A still faults at the same spot
-- because it enters 32-bit PM and our reflection is 16-bit-only.
The RM callback stubs end in `CF` which is 16-bit IRET; a 32-bit gate
pushes a 12-byte frame that can't be unwound by a 16-bit IRET.

Next step: 32-bit PM reflection via per-vector CB_IRETD shims
(stub = `FE 38 xx xx 66 CF`), one per vector, installed in the 32-bit
PM IDT.  That should unblock DOS32A.

[Update same day] 32-bit reflection landed.  DOS32A now completes
the PM switch and runs silently in 32-bit PM (no more fault).  Without
a client binary to load, it appears to sit in a wait loop -- the
remaining gap is not dosemu's but "give DOS32A something to do".
See dpmi_plan.md's "DOS32A probe after reflection" section for the
detailed picture and next-session pick.
